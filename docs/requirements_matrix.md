1. Матриця простежуваності вимог (RTM)

ID Вимоги

Вимога (Опис)

Функціональні Вимоги (FR)



FR-001

Користувач повинен мати можливість створити нове завдання, вказавши назву та опис.

FR-002

Система повинна автоматично встановлювати статус "todo" для нових завдань.

FR-003

Користувач повинен мати можливість отримати повний список усіх завдань.

FR-004

Користувач повинен мати можливість отримати дані одного конкретного завдання за його ID.

FR-005

Користувач повинен мати можливість оновити будь-яке поле існуючого завдання (назву, опис, статус).

FR-006

Користувач повинен мати можливість видалити існуюче завдання.

FR-007

Система повинна повертати помилку 404, якщо користувач намагається отримати, оновити або видалити неіснуюче завдання.

Нефункціональні Вимоги (NFR)



NFR-001

Усі відповіді API (окрім першого "холодного" запуску) повинні повертатися менш ніж за 500 мс.

NFR-002

Код бекенду повинен бути захищений від SQL-ін'єкцій (вирішено використанням SQLAlchemy ORM).

NFR-003

API не повинен приймати некоректні дані (наприклад, завдання без назви) (вирішено валідацією Pydantic).

NFR-004

Секрети (пароль до БД) не повинні зберігатися у коді (вирішено використанням .env файлу).

Вимоги до Тестування (TR)



TR-001

Код бекенду повинен бути покритий unit-тестами.

TR-002

API-ендпоінти повинні бути покриті інтеграційними тестами.

TR-003

Код повинен проходити статичний аналіз (Ruff).

TR-004

Покриття коду тестами має бути виміряне (pytest-cov).

TR-005

Усі тести та аналізи повинні автоматично запускатися на сервері CI/CD (GitHub Actions) при кожному push.
